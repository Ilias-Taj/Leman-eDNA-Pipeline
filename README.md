# Automated Bioinformatics eDNA Analysis for MinION

This repository contains a small, reproducible, project-local pipeline that:

- takes raw MinION reads (per-sample FASTQ) and Sanger references,
- trims adapters/primers and filters reads,
- maps reads to a Sanger reference database,
- produces a species-by-sample matrix of relative abundances,
- builds consensus sequences for well-supported targets and validates them against the Sanger DB.

This README explains what the pipeline does, the files it creates, how to run it (theory + examples), and how to interpret and troubleshoot results.

## Contents / files you should know about

- `2_run_preprocessing.py` - trim primers/adapters (cutadapt) and filter reads (filtlong/NanoFilt); outputs `out/<sample>/filtered_reads.fastq.gz`.
- `3_run_alignment.py` - map filtered reads to `refs/sanger_db.fa` using `minimap2` (writes `out/<sample>/alignments.paf`).
- `scripts/4_generate_matrix.py` - parse PAF best-hits into `out/<sample>/species_counts.csv`.
- `5_run_validation.py` - for each reference with enough high-quality mappings, extract supporting reads, build consensus (minimap2 + samtools + bcftools), align consensus back to DB and write `out/validation/<sample>/validation_report.csv`.
- `scripts/run_full_pipeline.sh` - convenience wrapper that runs preprocessing, alignment, matrix-generation and validation for all samples.
- `refs/sanger_db.fa` - the concatenated Sanger reference FASTA used as the mapping / validation DB.

Output directories (generated by runs):

- `out/<sample>/` - per-sample artifacts: `filtered_reads.fastq.gz`, `alignments.paf`, `species_counts.csv`.
- `out/validation/<sample>/` - per-sample validation artifacts and `validation_report.csv`.
- `out/merged/` - merged outputs: `species_by_sample.csv` (matrix) and `comparison_report.csv` (per-sample concordance summary).
- `out/logs/` - pipeline log files (global and per-sample logs).

## Tools and environment (what you need)

The pipeline is designed to run with a project-local conda prefix (example: `./env`). The essential command-line tools are:

- Python 3 (scripts use the standard library + built-in modules)
- `cutadapt` (adapter/primer trimming)
- `filtlong` or `NanoFilt` (read filtering)
- `minimap2` (mapping)
- `samtools` (SAM/BAM handling)
- `bcftools` (mpileup/call and `bcftools consensus`)
- `gzip`
You can create a project-local environment with something like (example using conda-forge packages):

```bash
conda create -p ./env -c conda-forge -y \
  python=3.10 cutadapt filtlong minimap2 samtools bcftools blast
```

If you already have `./env` created, run scripts with `conda run -p ./env <command>` to ensure they use the correct tools.

## High-level pipeline steps (theory)

1. Preprocessing
   - Trim primers/adapters from reads (single-end amplicon trimming; script handles cases where trimmed output is FASTA by converting to FASTQ with dummy qualities).
   - Filter reads by length and quality to keep the best amplicon reads.
   - Output: `out/<sample>/filtered_reads.fastq.gz`.

2. Alignment
   - Map filtered reads to the Sanger DB with `minimap2` (ONT preset `-x map-ont`), write PAF of mappings.
   - Output: `out/<sample>/alignments.paf`.

3. Matrix generation
   - Parse the PAF and assign each read to its best-hit reference; count reads and compute relative abundance and presence/absence for every reference.
   - Output (per sample): `out/<sample>/species_counts.csv`.

4. Consensus-based validation
   - For each reference with enough high-quality mappings (configurable `--mapq` and `--min_reads`), extract reads assigned to that reference, align reads to the reference, call variants and create a consensus (samtools + bcftools). Align the consensus back to the full DB and record the best hit and percent identity.
   - Output (per sample): `out/validation/<sample>/validation_report.csv` plus consensus and intermediate BAM/VCF files in `out/validation/<sample>/work_<ref>/`.

5. Merge and compare
   - Merge per-sample `species_counts.csv` files into a single `out/merged/species_by_sample.csv` matrix. Create `out/merged/comparison_report.csv` that summarizes per-sample top hits vs the expected Sanger id and reports validation outcome.

## What files are created at each step (detailed)

- Preprocessing (`2_run_preprocessing.py`)
  - `out/<sample>/trimmed.fasta` or `out/<sample>/trimmed.fastq` (intermediate)
  - `out/<sample>/filtered_reads.fastq.gz` (final)

- Alignment (`3_run_alignment.py`)
  - `out/<sample>/alignments.paf`

- Matrix generation (`scripts/4_generate_matrix.py`)
  - `out/<sample>/species_counts.csv` (columns: `reference_id,read_count,relative_abundance,presence`)

- Validation (`5_run_validation.py`)
  - `out/validation/<sample>/<ref>.reads.fastq` (extracted supporting reads)
  - `out/validation/<sample>/<ref>.ref.fa` (single-target reference)
  - `out/validation/<sample>/work_<ref>/aln.sorted.bam` and index
  - `out/validation/<sample>/work_<ref>/calls.vcf.gz`
  - `out/validation/<sample>/work_<ref>/consensus.fa`
  - `out/validation/<sample>/validation_report.csv`

- Merge / final
  - `out/merged/species_by_sample.csv` (matrix, references × samples)
  - `out/merged/comparison_report.csv` (per-sample concordance summary)

## How to run the full pipeline (examples)

First - theory: the wrapper `scripts/run_full_pipeline.sh` iterates samples, calls the main scripts (`2_run_preprocessing.py`, `3_run_alignment.py`, `scripts/4_generate_matrix.py`) and `5_run_validation.py` for each sample, and writes per-sample logs. Running it yields the per-sample outputs and the merged outputs described above.

Concrete example - run the full pipeline in the background (recommended for long runs):

```bash
# make logs directory
mkdir -p out/logs

# run wrapper via nohup so it survives terminal restarts
nohup bash scripts/run_full_pipeline.sh > out/logs/run_full_pipeline.log 2>&1 &

# check progress
tail -n 200 out/logs/run_full_pipeline.log
```

If you prefer to run step-by-step for a single sample, replace `<sample>` and use the scripts directly. Example (use `conda run -p ./env` to use the project env):

```bash
# 1) Preprocess
conda run -p ./env python3 scripts/2_run_preprocessing.py \
  --input_file "raw_reads/<sample>/assembly/raw.reads.unsorted" \
  --output_dir out/<sample> \
  --primer_f CGCCTGTTTATCAAAAACAT --primer_r CCGGTTTGAACTCAGATCA \
  --min_length 100 --keep_percent 90 --threads 4

# 2) Alignment
conda run -p ./env python3 scripts/3_run_alignment.py \
  --reads_file out/<sample>/filtered_reads.fastq.gz \
  --db_file refs/sanger_db.fa \
  --output_dir out/<sample> --threads 4

# 3) Matrix generation
conda run -p ./env python3 scripts/4_generate_matrix.py \
  --paf out/<sample>/alignments.paf --output out/<sample>/species_counts.csv

# 4) Validation (consensus)
conda run -p ./env python3 scripts/5_run_validation.py \
  --paf_file out/<sample>/alignments.paf \
  --reads_file "raw_reads/<sample>/assembly/raw.reads.unsorted" \
  --db_file refs/sanger_db.fa \
  --output_dir out/validation/<sample> --mapq 40 --min_reads 20 --threads 4

# 5) Merge summary (after running all samples)
python3 scripts/4_merge_and_compare.py
```

Notes:
- Use the `--mapq` and `--min_reads` options to control which reference targets get consensus-built during validation.
- If your input is FASTA (trimmed), the preprocessing script will convert it to FASTQ with dummy quality scores to keep the pipeline consistent.

## Interpreting the main final files

- `out/merged/species_by_sample.csv` - each row is a reference; columns are samples; values are relative abundance (0–1). Good for heatmaps and abundance tables.
- `out/merged/comparison_report.csv` - one row per sample summarizing: expected Sanger id, top MinION reference, top read counts and relative abundance, whether top MinION string matches the Sanger id, validation_status and consensus identity percentage.

Common validation_status values:
- `PASS` - consensus built and its best DB alignment was accepted.
- `TOO_FEW_READS` - not enough high-MAPQ reads mapped to the target to build a consensus.
- `NO_ALIGN` - consensus built but it did not align to the Sanger DB (possible issue with consensus or DB mismatch).
- `CONS_FAIL` - consensus-building failed (VCF/BCFtools step failed).

## Troubleshooting checklist

- If `out/<sample>/filtered_reads.fastq.gz` is empty or very small:
  - Check per-sample log in `out/logs/<sample>.log` for cutadapt messages (paired-end flags, primer not found, etc.).
  - Ensure correct primer sequences are provided for that sample.

- If `alignments.paf` is empty or maps very few reads:
  - Check that `filtered_reads.fastq.gz` contains reads and that mapping preset (`-x map-ont`) is appropriate.
  - Verify `refs/sanger_db.fa` contains relevant sequences.

- If `validation_report.csv` shows `NO_ALIGN` or very low identity:
  - Inspect consensus at `out/validation/<sample>/work_<ref>/consensus.fa`.
  - Re-align the consensus to the DB manually: `conda run -p ./env minimap2 -c refs/sanger_db.fa consensus.fa` or `blastn` for more detail.

- If many references are `TOO_FEW_READS`:
  - Consider lowering `--min_reads` (but accept that lower read counts reduce consensus reliability), or relax `filtlong` trimming thresholds.

## Example interpretation (based on a completed run)

- `out/merged/species_by_sample.csv` shows clear dominant references per sample. The `comparison_report.csv` summarizes whether the MinION dominant hit agrees with the Sanger id and whether the consensus validation passed.
- Look at the `validation_identity_pct` value to judge confidence - >90% is usually strong for marker sequences; lower values require manual inspection.

## Reproducibility notes

- All CLI tools should be installed into a project-local conda prefix (e.g., `./env`) to make the pipeline reproducible. Use `conda run -p ./env ...` when calling scripts or place the env `bin/` directory at the front of your `PATH` in your shell.
